# Codex Implementation Handoff: Flower Recommendation Web App

Build a polished, responsive flower recommendation web application for local development.

The application helps customers browse flowers visually, identify what characteristics they prefer, receive matching recommendations, and create a summary they can show to a florist.

The key product transformation is:

**Simple visual choices become useful language that a florist can act on.**

# Product objective

Customers often know which flowers they find attractive but cannot easily describe:

* The flower names
* The colors they prefer
* The petal shapes they like
* The arrangement style they want
* The right flowers for an occasion
* The symbolic meanings associated with flowers

The application should let customers browse individual flowers and arrangements, then produce:

1. A summary of the characteristics they appear to prefer
2. Recommended flowers or arrangements
3. A short hanakotoba-inspired message
4. A concise summary they can show to the flower shop owner

The experience should assist a florist, not replace professional advice.

# Where to obtain flower photographs

Use sources in this priority order.

## Option 1: Photographs from the flower shop

This is the best option.

Ask the owner to provide photographs of actual inventory.

Benefits:

* The flowers match what customers can buy.
* Prices and appearance remain connected.
* The prototype feels specific to the shop.
* There is less ambiguity about cultivars.
* The owner can confirm permission to use the images.

For each image, record:

```text
filename
flower ID
photographer or owner
permission status
date received
```

Recommended photograph requirements:

* One main flower or arrangement
* Portrait or near-square crop
* Minimal background clutter
* No embedded text
* At least 1200 pixels on the shortest useful dimension
* Similar lighting and composition where possible

Do not use screenshots from Instagram, Pinterest, Google Images, other florists, or flower marketplaces unless the rights holder has explicitly authorized reuse.

## Option 2: Pexels for temporary prototype images

Pexels can be used to fill gaps in a local prototype. Its published license allows photos to be used without required attribution, although preserving photographer and source information is still good practice.

Search manually for terms such as:

```text
pink ranunculus flower
white lisianthus
yellow tulip close up
red rose single bloom
purple hydrangea
sunflower bouquet
pink peony
white orchid
```

Download the selected images manually.

Do not add a Pexels API to the application.

Record:

```text
image_source_url
photographer
download date
```

Be aware that stock-photo search results may not identify exact varieties reliably. The shop owner should verify that the image reasonably represents the named flower.

## Option 3: Wikimedia Commons

Wikimedia Commons is useful when you need a photograph of a specific species.

Every file can have its own license and attribution requirements. Use the file page or MediaWiki API metadata to record the creator, source, and license.

For every selected image, save:

```text
original file page
creator
license name
license URL
whether the image was cropped or modified
```

Do not assume that all Wikimedia images use the same license.

## Option 4: Openverse

Openverse searches openly licensed and public-domain media from multiple sources.

Use it as a search tool, but verify the license on the original source page before including an image. Openverse states that it does not control the source content or guarantee the accuracy of every licensing record.

For a short hackathon, manual verification is preferable to building an Openverse API integration.

## Avoid the Unsplash API for this MVP

The Unsplash API requires applications to use the image URLs returned by the API directly and follow its hotlinking and attribution rules.

Since this project should use local static images and no runtime data services, do not add the Unsplash API.

# Recommendation logic

Use deterministic client-side scoring.

Do not call an LLM.

## Preference aggregation

For each liked flower:

* Add one point for each color.
* Add one point for each tag.
* Add one point for each shape.
* Add one point for each style.

For each favorite flower:

* Apply the normal liked score.
* Add two additional points to its traits.

Return:

* Top three colors
* Top three tags
* Top two shapes
* Top two styles

Use stable alphabetical ordering to resolve equal scores.

## Recommendation scoring

Score unselected flowers:

```text
+3 matching top style
+2 matching top color
+2 matching top shape
+1 matching top tag
+1 appropriate for selected occasion
+1 within selected budget
```

Exclude explicitly skipped flowers unless there are too few candidates.

Return three to five recommendations.

Keep this implementation readable and unit-tested.

# Performance

The application should:

* Use static local assets.
* Make no runtime API requests.
* Render at most three discovery cards.
* Load only the active and next card images eagerly.
* Lazy-load later images.
* Use explicit image dimensions or aspect ratios.
* Avoid layout shifts.
* Avoid excessive animation.
* Avoid unnecessary dependencies.

Example image:

```tsx
<img
  src={flower.image.srcSmall ?? flower.image.src}
  srcSet={
    flower.image.srcSmall
      ? `${flower.image.srcSmall} 480w, ${flower.image.src} 720w`
      : undefined
  }
  sizes="(max-width: 600px) 100vw, 520px"
  loading={isActive || isNext ? "eager" : "lazy"}
  fetchPriority={isActive ? "high" : "auto"}
  width={720}
  height={900}
  alt={flower.image.alt}
/>
```
